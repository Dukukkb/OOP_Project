# Система управления задачами

## Этап 1. Проектирование доменной области

## 1. Описание предметной области

Проект представляет собой систему управления проектами и задачами.

Основная задача приложения - позволить пользователям создавать проекты, добавлять участников, распределять задачи между исполнителями, отслеживать статусы выполнения и обсуждать задачи через комментарии.

Система помогает команде видеть общий список задач, понимать, кто за что отвечает, какие задачи уже выполнены, а какие находятся в работе.

## 2. Бизнес-контекст

Команда работает над проектом и хочет удобно распределять задачи между участниками.

Например, пользователь создает проект "Проект по ООП", добавляет туда участников команды, создает задачу "Разработать API-контракты", назначает исполнителя и указывает срок выполнения.

После этого исполнитель может менять статус задачи, а остальные участники могут оставлять комментарии для обсуждения деталей.

## 3. Основные сценарии использования

### 3.1 Регистрация пользователя

Пользователь создает учетную запись в системе.

**Основной сценарий:**

1. Пользователь вводит имя, email и пароль.
2. Система проверяет корректность данных.
3. Система создает пользователя.
4. Пользователь получает возможность создавать проекты и участвовать в них.

---

### 3.2 Авторизация пользователя

Пользователь входит в систему, чтобы получить доступ к проектам и задачам.

**Основной сценарий:**

1. Пользователь вводит email и пароль.
2. Система проверяет введенные данные.
3. Если данные верны, система авторизует пользователя.
4. Пользователь получает доступ к своим проектам и задачам.

---

### 3.3 Создание проекта

Пользователь создает новый проект.

**Основной сценарий:**

1. Пользователь вводит название проекта.
2. Пользователь добавляет описание проекта.
3. Пользователь указывает дату начала и дату окончания.
4. Система создает проект.
5. Пользователь становится владельцем проекта.

---

### 3.4 Редактирование и удаление проекта

Пользователь может изменить данные проекта или удалить проект, если он больше не нужен.

**Основной сценарий редактирования:**

1. Пользователь выбирает проект.
2. Пользователь изменяет название, описание, даты или статус.
3. Система проверяет корректность данных.
4. Система сохраняет обновленный проект.

**Основной сценарий удаления:**

1. Пользователь выбирает проект.
2. Система проверяет, есть ли в проекте незавершенные задачи.
3. Если незавершенных задач нет, система удаляет проект.

---

### 3.5 Добавление участника в проект

Владелец проекта добавляет пользователя в проект.

**Основной сценарий:**

1. Владелец проекта выбирает проект.
2. Владелец указывает пользователя, которого нужно добавить.
3. Владелец выбирает роль пользователя в проекте.
4. Система добавляет пользователя в список участников проекта.

---

### 3.6 Создание задачи

Пользователь создает задачу внутри проекта.

**Основной сценарий:**

1. Пользователь выбирает проект.
2. Пользователь вводит название задачи.
3. Пользователь добавляет описание задачи.
4. Пользователь указывает приоритет.
5. Пользователь указывает срок выполнения.
6. Пользователь назначает исполнителя.
7. Система создает задачу со статусом `NEW`.

---

### 3.7 Обновление задачи

Пользователь может изменить данные задачи.

**Основной сценарий:**

1. Пользователь выбирает задачу.
2. Пользователь изменяет название, описание, исполнителя, приоритет, срок или статус.
3. Система проверяет корректность данных.
4. Система сохраняет обновленную задачу.

---

### 3.8 Изменение статуса задачи

Исполнитель меняет статус задачи во время работы.

**Основной сценарий:**

1. Исполнитель открывает задачу.
2. Исполнитель выбирает новый статус.
3. Система проверяет допустимость статуса.
4. Система сохраняет новый статус задачи.

---

### 3.9 Комментирование задачи

Пользователь оставляет комментарий к задаче.

**Основной сценарий:**

1. Пользователь открывает задачу.
2. Пользователь вводит текст комментария.
3. Система проверяет, что комментарий не пустой.
4. Система сохраняет комментарий.

---

### 3.10 Просмотр списка задач проекта

Участник проекта просматривает список задач.

**Основной сценарий:**

1. Пользователь открывает проект.
2. Система показывает список задач проекта.
3. Пользователь видит статусы, исполнителей, приоритеты и сроки выполнения.
4. При необходимости пользователь фильтрует задачи по статусу или исполнителю.

## 6. Связи между сущностями

| Связь                    | Описание                                             |
| ------------------------ | ---------------------------------------------------- |
| User 1 -> N Project      | Один пользователь может создать много проектов       |
| Project 1 -> N Task      | Один проект может содержать много задач              |
| User 1 -> N Task         | Один пользователь может создать много задач          |
| User 1 -> N Task         | Один пользователь может быть исполнителем задач      |
| Project N -> M User      | Один проект имеет много участников                   |
| Task 1 -> N Comment      | Одна задача может иметь много комментариев           |
| User 1 -> N Comment      | Один пользователь может оставить много комментариев  |

## 7. Ограничения

1. Email пользователя должен быть уникальным.
2. Пароль пользователя не может быть пустым.
3. Проект обязан иметь владельца.
4. Название проекта не может быть пустым.
5. Дата окончания проекта должна быть позже даты начала.
6. Задача должна принадлежать существующему проекту.
7. Название задачи не может быть пустым.
8. Исполнитель задачи должен быть участником проекта.
9. Дедлайн задачи должен быть позже даты создания.
10. Статус задачи может принимать только допустимые значения.
11. Приоритет задачи может принимать только допустимые значения.
12. Комментарий не может быть пустым.
13. Проект нельзя удалить, если в нем есть незавершенные задачи.

---

## 8. Бизнес-правила

1. Каждый проект имеет одного владельца.

2. Пользователь может быть участником нескольких проектов.

3. В одном проекте может быть несколько участников.

4. Роль пользователя внутри проекта хранится отдельно от общей роли пользователя в системе.

5. Задача создается только внутри существующего проекта.

6. Задача может иметь одного автора и одного исполнителя.

7. Исполнителем задачи может быть только пользователь, который состоит в проекте.

8. При создании задачи ей автоматически присваивается статус `NEW`.

9. Статус задачи может изменяться по мере выполнения работы.

10. Если задача завершена, она получает статус `DONE`.

11. Комментарии используются для обсуждения конкретной задачи.

12. Удаление проекта запрещено, если в проекте есть незавершенные задачи.

## Этап 2. Проектирование API и контрактов

## 1. Общая схема взаимодействия

В рамках проекта используется монолитная backend-архитектура с логическим разделением на сервисы.

Клиент взаимодействует с одним backend-приложением через REST API. Внутри приложения запросы обрабатываются отдельными сервисами: User Service, Project Service, Task Service и Comment Service.

```text
      Client
        |
        v
 Backend REST API
        |
        |--- User Service
        |--- Project Service
        |--- Task Service
        |--- Comment Service
```

## 2. Сервисы системы

## 2.1 User Service

User Service отвечает за работу с пользователями.

Основные задачи:

- регистрация пользователя
- авторизация пользователя
- получение данных текущего пользователя
- получение списка пользователей

## 2.2 Project Service

Project Service отвечает за работу с проектами и участниками проектов.

Основные задачи:

- создание проекта
- получение списка проектов
- получение одного проекта
- обновление проекта
- удаление проекта
- добавление участника в проект
- получение участников проекта
- удаление участника из проекта

## 2.3 Task Service

Task Service отвечает за работу с задачами.

Основные задачи:

- создание задачи
- получение списка задач проекта
- получение одной задачи
- обновление задачи
- изменение статуса задачи
- удаление задачи

## 2.4 Comment Service

Comment Service отвечает за комментарии к задачам.

Основные задачи:

- добавление комментария
- получение комментариев задачи
- удаление комментария

## 3. Общие правила API

Все данные передаются в формате JSON.

Все ручки, кроме регистрации и авторизации, требуют авторизации пользователя.

Для защищенных запросов используется заголовок:

```http
Authorization: Bearer jwt-token
```

Статус задачи может принимать только следующие значения:

```text
NEW
IN_PROGRESS
REVIEW
DONE
```

Приоритет задачи может принимать только следующие значения:

```text
LOW
MEDIUM
HIGH
```

## 9. Единый формат ошибок

Для всех сервисов используется единый формат ошибки.

```json
{
  "statusCode": 400,
  "error": "Bad Request",
  "message": "Название задачи не может быть пустым"
}
```

Основные коды ошибок:

| Код | Название              | Когда используется                            |
| --- | --------------------- | --------------------------------------------- |
| 400 | Bad Request           | Некорректные данные в запросе                 |
| 401 | Unauthorized          | Пользователь не авторизован                   |
| 403 | Forbidden             | Недостаточно прав для выполнения операции     |
| 404 | Not Found             | Объект не найден                              |
| 409 | Conflict              | Конфликт бизнес-правил                        |
| 500 | Internal Server Error | Непредвиденная ошибка сервера                 |
